# 从零逆向一个医学考试APP的全套加密体系

> 记一次对 uni-app Android 应用的完整逆向工程。从 APK 解包到 AES 密钥破解，12个 Frida Hook 版本的迭代，最终用纯 Python 实现全部 API 调用。

---

## 背景

目标 APP 是一个医学考试刷题应用，Web 版功能不全。为了在电脑上使用完整功能，决定逆向其 API 并自己实现客户端。

抓包后发现：请求头有 AES 加密签名，响应体也是全加密的。这不是简单的 Token 鉴权，而是**完整的端到端加密**。

---

## 第一阶段：侦察

### 1.1 APK 结构分析

用 `apktool d target.apk` 解包后，看到了关键的目录结构：

```
assets/
  apps/
    __UNI__7EE4208/
      www/
        app-service.js    ← 3.6MB, 全部业务逻辑
        manifest.json     ← 应用配置
```

这是 **DCloud uni-app** 框架 —— 基于 Vue.js 的跨平台方案。业务逻辑全在一个巨大的 JavaScript 文件里，跑在 Android 的 Weex V8 引擎上。

这个判断非常关键，因为它决定了后续所有的 Hook 策略。

### 1.2 初步抓包

请求格式：
```http
POST /api/login HTTP/1.1
AppVersion: 4.1.2
AppVerify: aes:zxT8cR4uugV1xcMr0ikZKiWBOCwAn//tkpv0XpsW1EG...
Content-Type: application/json

{"phone":"138****1234","password":"****"}
```

响应格式：
```json
{
  "code": 200,
  "msg": "操作成功",
  "encryptData": "CXO6W2LtBbq2JHz...约1000字符的Base64...",
  "encryptSign": "db8778dd987c95fe4edf7917dcc4a4cf"
}
```

要调用 API，需要解决：
1. 怎么生成 `AppVerify`（请求签名）
2. 怎么解密 `encryptData`（响应数据）

这两个都需要找到 AES 的 Key 和 IV。

## 第二阶段：静态分析 —— 破解混淆

### 2.1 混淆结构

`app-service.js` 使用了经典的字符串数组 + 查表函数混淆：

```javascript
// 字符串数组 (约11000个元素)
function a1_0x1557() {
    var _0x = ['AES', 'encrypt', 'MD5', 'yhyk2026xzdlcxwc', ...];
    a1_0x1557 = function() { return _0x; };
    return _0x;
}

// 查表函数 (带偏移)
function a1_0x43dc(_0x4a71, _0x43dc) {
    var _0x1557 = a1_0x1557();
    _0x4a71 -= 0x7e;
    return _0x1557[_0x4a71];
}

// 实际代码中的引用
_0x1a0371['default'][a1_0x43dc(0x652)]  // → CryptoJS.AES
```

### 2.2 构建解码器

关键洞察：查表函数本身是合法的 JavaScript，可以在 Node.js 中直接执行：

```javascript
// decode_obfuscation.js
const data = fs.readFileSync('app-service.js', 'utf8');

// 提取并 eval 查表函数
eval(extractFunction(data, 'a1_0x1557') + '\n' + 
     extractFunction(data, 'a1_0x43dc') + '\n' + 
     extractShuffleCode(data));

// 解码任意 hex 值
console.log(a1_0x43dc(0x652));   // → "AES"
console.log(a1_0x43dc(0x25ba));  // → "yhyk2026xzdlcxwc"
console.log(a1_0x43dc(0x1593));  // → "build_query"
```

用这个方法批量解码了所有关键值：

| Hex | 解码值 | 用途 |
|-----|--------|------|
| `0x652` | `AES` | 加密算法 |
| `0x1ea1` | `encrypt` | 加密方法 |
| `0x2c0` | `decrypt` | 解密方法 |
| `0x25ba` | `yhyk2026xzdlcxwc` | 默认 IV (16字节!) |
| `0x1b8c` | `Pkcs7` | 填充模式 |
| `0x1593` | `build_query` | 签名构建函数 |
| `0x15fd` | `md5` | 哈希函数 |
| `0xdf9` | `substr` | 字符串截取 |

### 2.3 还原 AppVerify 加密

在解码的代码中追踪加密调用链：

```javascript
// 反混淆还原后的关键代码:
key = CryptoJS.MD5('AppVerify').toString(Hex).substr(7, 16);
// → MD5("AppVerify") = "d7cfba8cfd0f9faab3726e5c03da22f1"
//                            ^^^^^^^^^^^^^^^^
//                       [7:23] = "cfd0f9faab3726e5"   ← AES Key!

iv = "yhyk2026xzdlcxwc";                               // ← AES IV!

encrypted = CryptoJS.AES.encrypt(
    JSON.stringify(avData), 
    CryptoJS.enc.Utf8.parse(key),
    { iv: CryptoJS.enc.Utf8.parse(iv), mode: CryptoJS.mode.CBC, padding: CryptoJS.pad.Pkcs7 }
);
```

验证 —— 用 Python 解密捕获的 AppVerify：

```python
from Crypto.Cipher import AES
from Crypto.Util.Padding import unpad
import base64, json

KEY = b'cfd0f9faab3726e5'
IV  = b'yhyk2026xzdlcxwc'

captured = "zxT8cR4uugV1xcMr0ikZKiWBOCwAn//tkpv0XpsW1EG..."
ct = base64.b64decode(captured)
plain = unpad(AES.new(KEY, AES.MODE_CBC, IV).decrypt(ct), 16)
print(json.loads(plain))
```

输出：
```json
{
  "route": "pages/login/login",
  "random": "QMBVdcwHFdeqBLrViXWQUQtEROMrizbj",
  "time": 1775295342.683,
  "sign": "49723cac683fa8d8aa77640359b6a6e5",
  "appId": "__UNI__7EE4208",
  "appSign": "e65425ed05100e7ead3b9fda147e3b31c2c8c7d9"
}
```

**第一把钥匙到手。**

## 第三阶段：动态分析 —— Frida Hook 迭代

### 3.1 框架层的坑 (v1-v4)

uni-app 不是标准 WebView，加密逻辑跑在 DCloud 的 **Weex V8 引擎**里。

| 版本 | 方法 | 结果 |
|------|------|------|
| v1 | Hook OkHttp Interceptor | ❌ uni-app 不走 OkHttp 拦截器 |
| v2 | Hook String 构造函数 | ❌ V8引擎频繁调用，APP卡死 |
| v3 | Hook OkHttp addHeader | ❌ 请求头已经是加密后的 |
| v4 | 延迟注入 (setTimeout 12s) | ✅ 解决启动阶段崩溃 |

**教训**：逆向不只是看代码，还要理解框架的请求链路。

### 3.2 找到正确的 Hook 点 (v5-v7)

通过分析 DCloud SDK 源码，找到了关键路径：

```
JS (Weex V8) → WXStreamModule.fetch() → DCWXHttpAdapter → OkHttp
```

最终有效的 Hook 点是 `DCWXHttpAdapter.getOKRequest`：

```javascript
// hook_capture.js (核心部分)
var Adapter = Java.use("io.dcloud.feature.weex.adapter.DCWXHttpAdapter");
Adapter.getOKRequest.implementation = function(builder, wxReq, listener) {
    var okReq = this.getOKRequest(builder, wxReq, listener);
    var url = okReq.url().toString();
    if (url.indexOf("yhykwch") !== -1) {
        var headers = okReq.headers();
        // 完整的 AppVerify 和 Token 都在这里!
        console.log("[AppVerify] " + headers.get("AppVerify"));
        console.log("[Token] " + headers.get("Token"));
    }
    return okReq;
};
```

响应体通过 Hook `readInputStreamAsBytes` 获取：

```javascript
Adapter.readInputStreamAsBytes.implementation = function(is, l) {
    var r = this.readInputStreamAsBytes(is, l);
    var s = Java.use("java.lang.String").$new(r, "UTF-8").toString();
    if (s.indexOf('"encryptData"') !== -1) {
        console.log("[RESP] " + s);
    }
    return r;
};
```

### 3.3 配对验证法 (v12)

最终版 Hook 同时捕获**请求和响应的配对数据**，然后用 Python 分析器验证算法：

```
frida -U -f uni.UNI7EE4208 -l hook_capture.js | tee capture.txt
python analyze_capture.py capture.txt
```

这种"先抓数据，再验证算法"的方法非常高效——如果能复现所有捕获数据的加解密结果，算法就是对的。

## 第四阶段：破解响应解密

### 4.1 嵌入式 IV

代码分析发现了一个精妙的设计 —— **IV 被嵌入在密文字符串中间**：

```javascript
// 反混淆还原:
ivPosition = (encryptData.length - 24) % 128;
embeddedIV = encryptData.substr(ivPosition, 24);    // 24字符 = 18字节 Base64
ciphertext = encryptData 去掉这24个字符;
```

IV 的位置取决于数据长度，每次响应都不同。不看代码根本猜不到。

### 4.2 动态密钥派生

解密密钥不是固定的，而是从请求参数派生：

```javascript
keySource = request.random + embeddedIV + unique_code;
key = MD5(keySource).substr(7, 16);    // 16字符 = AES-128
iv  = MD5(embeddedIV).substr(11, 16);
```

这意味着：每个请求用不同的 `random`，所以每个响应的解密密钥都不同。

Python 实现：

```python
def decrypt_response(encrypt_data, request_random, unique_code):
    iv_pos = (len(encrypt_data) - 24) % 128
    emb_iv = encrypt_data[iv_pos:iv_pos + 24]
    ct_b64 = encrypt_data[:iv_pos] + encrypt_data[iv_pos + 24:]
    
    ks = request_random + emb_iv + unique_code
    key = md5(ks)[7:23].encode()
    iv  = md5(emb_iv)[11:27].encode()
    
    plain = unpad(AES.new(key, AES.MODE_CBC, iv).decrypt(base64.b64decode(ct_b64)), 16)
    return json.loads(plain)
```

## 第五阶段：最后的三个坑

算法全部还原了，但服务端返回"签名错误"。花了最多时间的阶段。

### 坑1：Token 来源的错位

FETCH-OPTS（JS 层的 fetch 参数）里**没有 Token 头**，但 OkHttp 层的请求**有 Token 头**。

这说明 Token 在 Java 层添加，而 JS 层的 `takeToken()` 返回的是空值。所以 `build_query` 里的 token 字段：

```
登录前: token = "tourist:" + random  (因为 takeToken() 为空)
登录后: token = DCStorage 中存储的加密 token (不是 OkHttp 的 Token 头!)
```

关键区别：`build_query` 用的 token 和 HTTP 头里的 Token 是**同一个值**，但它们通过不同的渠道获得。

### 坑2：unique_code 的计算

这个值不在任何请求/响应中出现，是客户端本地计算的：

```javascript
unique_code = MD5(appId + "-yhyk2026-" + randomKey)
// randomKey 为空时:
// MD5("__UNI__7EE4208-yhyk2026-") = "fe514637e1eb8c298e3e5c2b35fbb786"
```

这个值同时用在 sign 计算和响应解密两个地方。它在早期代码分析中出现过，但一直没意识到需要用在 sign 里。

### 坑3：时间戳精度

JavaScript 的 `Date.now()/1000` 最多3位小数（因为 `Date.now()` 返回整数毫秒），但 Python 的 `time.time()` 有6位以上。

```python
# ❌ 错误: Python 默认精度
time.time()              # 1775350757.269701

# ✅ 正确: 匹配 JavaScript 精度
round(time.time(), 3)    # 1775350757.270
```

服务端从 AppVerify 中取出 `time` 字段重算 sign，精度不一致导致 sign 不匹配。

**三个坑同时存在，任何一个不对都是"签名错误"。这是最折磨人的部分——你不知道是哪个参数错了，还是全错了。**

### 加密体系总结

```

│ AppVerify 请求签名                                │
│   Key = MD5("AppVerify")[7:23]                   │
│   IV  = "yhyk2026xzdlcxwc"                       │

```

## 方法论总结

1. **先侦察再动手**。APK 解包后先判断技术栈（uni-app/React Native/Flutter/原生），这决定了后续所有策略。

2. **混淆 ≠ 安全**。字符串混淆只是增加阅读成本，查表函数本身可以被直接执行来反混淆。

3. **理解框架比理解业务更重要**。uni-app 的请求不走标准 WebView 也不走标准 OkHttp 拦截器。找到 `DCWXHttpAdapter` 花的时间比破解 AES 还多。

4. **配对验证法**。先用 Hook 抓到足够的配对数据（请求+响应），然后离线验证算法。不要试图一步到位。

5. **当多个参数同时错误时，不要怀疑方向**。sign 的三个坑（token来源、unique_code、时间精度）同时存在。修一个不够，会让你误以为整个方向都错了。保持信心，逐个排除。

6. **密码明文传输是更大的安全问题**。前端加密体系做得再复杂，登录接口还是明文传密码。安全应该在传输层（HTTPS）和服务端实现。
