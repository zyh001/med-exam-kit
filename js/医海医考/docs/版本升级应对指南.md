# 医海医考 APK 版本升级应对指南

> 当前已破解版本: **4.1.2** (versionCode 412)  
> 本文档帮助你在 APP 升级后快速判断哪些东西变了、怎么修复。

---

## 一、升级后什么可能变、什么不太会变

```
几乎不会变 (服务端决定)          可能会变 (前端决定)            大版本才会变
─────────────────────        ─────────────────────        ─────────────────
• API 路径 (/api/xxx)         • 混淆变量名                  • 加密算法 (AES→其他)
• 响应 JSON 结构              • 查表函数名/偏移量            • 请求签名体系重构
• 请求参数字段名              • AES Key 推导方式             • 响应加密架构变化
• Token 机制                  • unique_code 计算方式         • 通信协议 (REST→其他)
• 服务端域名                  • sign 参数新增/删减
                              • appSign (换签名证书)
                              • AppVersion 版本号
```

**经验法则**: 小版本升级 (4.1.2→4.1.3) 通常只改混淆和业务逻辑，加密体系不变。大版本 (4.x→5.x) 才可能重构加密。

## 二、快速诊断流程

拿到新版 APK 后，按这个顺序排查，通常10分钟内就能定位问题：

```
第1步: 直接用旧 SDK 试
  │
  ├─ code=200 → 没变，继续用
  │
  ├─ code=1011 (签名错误) → 第2步
  │
  └─ 连接失败/域名变了 → 第6步
  
第2步: 检查 AppVersion
  │
  ├─ 改了版本号 → 更新 SDK 里的 APP_VERSION，重试
  │    如果还是1011 → 第3步
  │
  └─ 版本号没变 → 第3步

第3步: 检查 AES 密钥
  │
  ├─ 解密旧版捕获的 AppVerify 还能用 → 密钥没变，第4步
  │
  └─ 解密失败 → 密钥变了，第5步

第4步: 检查 sign 参数
  │
  ├─ 参数增减 → 更新 build_query
  ├─ unique_code 算法变了 → 更新计算方式
  └─ appSign 变了 (换了签名证书) → 更新 APP_SIGN

第5步: 重新破解密钥
  │
  └─ 详见下文「密钥重新提取」

第6步: 检查域名/路径变化
  └─ 从新版 app-service.js 搜索 baseUrl
```

## 三、各步骤的具体操作

### 3.1 第1步：直接试旧 SDK

```python
from yhyk_sdk import YhykClient
c = YhykClient()
r = c.login("手机号", "密码")
print(r)
# code=200 → 没变
# code=1011 → 签名体系有变化
# code=1012 → 时间校验有变化
```

### 3.2 第2步：检查版本号

```bash
# 解包新 APK
apktool d new_version.apk -o new_apk

# 查看版本号
grep -r "versionName\|versionCode" new_apk/apktool.yml
```

SDK 中更新：
```python
APP_VERSION = '4.2.0'  # 改成新版本号
```

### 3.3 第3步：验证 AES 密钥是否还有效

用 Frida 抓一个新版的 AppVerify，然后用旧密钥尝试解密：

```python
from yhyk_sdk import decrypt_appverify

# 粘贴从新版抓到的 AppVerify
captured = "aes:zxT8cR4uugV1xcMr0ikZK..."
try:
    result = decrypt_appverify(captured)
    print("✅ 旧密钥仍然有效!")
    print(result)
except:
    print("❌ 密钥已变，需要重新提取")
```

### 3.4 第4步：检查 sign 参数变化

如果密钥没变但 sign 不匹配，说明 `build_query` 的参数有变化。

```bash
# 从新版 app-service.js 找 build_query 调用
node scripts/decode_obfuscation.js new_apk/assets/apps/__UNI__7EE4208/www/app-service.js 0x1593
# → 确认 build_query 函数名是否变了

# 查看 build_query 附近的代码
grep -oP ".{500}build_query.{500}" new_apk/assets/apps/__UNI__7EE4208/www/app-service.js > context.txt
# 用解码器处理 context.txt 中的 hex 值
```

常见变化：
- **新增参数**: sign 计算多了一个字段（如 `device_id`）
- **unique_code 算法变了**: 前缀或种子字符串改了
- **appSign 变了**: 用了新的签名证书

### 3.5 第5步：密钥重新提取

如果 AES 密钥变了，需要从新的 `app-service.js` 重新提取：

```bash
# 1. 先更新解码器（查表函数可能变了）
#    打开新的 app-service.js，找到 a1_0xXXXX 格式的函数名
grep -oP "function a1_0x[a-f0-9]+\(" new_app-service.js | head -5

# 2. 修改 decode_obfuscation.js 中的函数名

# 3. 搜索关键模式
```

密钥通常藏在这些地方：

```javascript
// 模式1: MD5 某个硬编码字符串后截取
key = MD5('SomeString').substr(offset, 16)

// 模式2: 直接硬编码
iv = 'some16charvalue!'

// 模式3: 多级推导
seed = 'yhyk' + year
key = MD5(MD5(seed) + salt).substr(7, 16)
```

搜索策略：
```bash
# 搜索 AES 相关调用
grep -oP "AES.{0,20}encrypt|AES.{0,20}decrypt" new_app-service.js

# 搜索 16 字符的硬编码字符串 (可能是新 IV)
grep -oP "'[a-zA-Z0-9]{16}'" new_app-service.js

# 搜索 MD5 + substr 模式
grep -oP "MD5.{0,100}substr" new_app-service.js

# 搜索 yhyk 相关的种子
grep -oP "yhyk[a-zA-Z0-9]*" new_app-service.js
```

### 3.6 第6步：域名/路径变化

```bash
# 搜索 API 域名
grep -oP "https://[a-z.]+yhyk[a-z.]*" new_app-service.js

# 搜索 baseUrl 配置
grep -oP "baseUrl.{0,100}" new_app-service.js
```

## 四、如果混淆方式完全变了

极端情况：换了混淆工具，查表函数结构不同。

### 4.1 判断方法

```bash
# 看是否还有 a1_0x 格式的函数
grep -c "a1_0x" new_app-service.js
# 如果 = 0，说明换了混淆方式

# 看有没有其他格式的查表函数
head -100 new_app-service.js
```

### 4.2 应对方案

| 新混淆类型 | 特征 | 破解思路 |
|-----------|------|---------|
| 字符串数组+查表 (同类) | `_0xABCD(0x123)` | 同样提取查表函数 eval |
| 字符串加密 (如 obfuscator.io) | `atob('base64')` | 搜索 atob/decode 批量解码 |
| 控制流平坦化 | `switch/case` 状态机 | 需要用 AST 工具还原 |
| WASM 加密 | `.wasm` 文件 | 需要逆向 WASM，难度大幅增加 |

### 4.3 万能方案：Frida 直接 Hook

无论混淆怎么变，只要加密还在 JS 层，Frida Hook 就能抓到明文：

```javascript
// 通用 CryptoJS Hook (不依赖混淆)
// 找到 CryptoJS 的 AES.encrypt 并 hook
Java.perform(function(){
    // Weex V8 引擎的 eval 注入
    var WXBridge = Java.use("com.taobao.weex.bridge.WXBridge");
    WXBridge.callNative.implementation = function(id, tasks, cb) {
        // 拦截 JS→Java 的所有调用
        var s = tasks.toString();
        if (s.indexOf("encrypt") !== -1 || s.indexOf("decrypt") !== -1) {
            console.log("[CRYPTO] " + s.substring(0, 500));
        }
        return this.callNative(id, tasks, cb);
    };
});
```

或者用最底层的 Hook —— 直接抓 HTTP 请求和响应：

```javascript
// DCWXHttpAdapter 是 DCloud 框架的常量，不受 JS 混淆影响
var A = Java.use("io.dcloud.feature.weex.adapter.DCWXHttpAdapter");
A.getOKRequest.implementation = function(b, r, l) {
    var req = this.getOKRequest(b, r, l);
    // 请求头在这里，一定能抓到
    console.log(req.headers().toString());
    return req;
};
```

**这个 Hook 点在 Java 层，不受 JS 混淆影响。只要 APP 还是 uni-app 框架就一定有效。**

## 五、版本升级检查清单

```
□ 1. 用旧 SDK 试一次 login
□ 2. 如果失败，apktool 解包新 APK
□ 3. 对比 apktool.yml 中的 versionName
□ 4. 更新 APP_VERSION，重试
□ 5. 如果还失败，Frida 抓一个新版 AppVerify
□ 6. 用旧 KEY/IV 尝试解密
□ 7. 如果能解密 → sign 参数变了，检查 build_query
□ 8. 如果不能解密 → 密钥变了，从新 JS 提取
□ 9. 搜索 MD5/AES/substr 模式定位新密钥
□ 10. 更新 SDK，验证全部接口
```

## 六、自动化监控（可选）

写一个定时任务，每天自动检测 SDK 是否还能用：

```python
#!/usr/bin/env python3
"""每天跑一次，检测 API 是否还能正常调用"""
from yhyk_sdk import YhykClient
import sys

c = YhykClient()
r = c.login("手机号", "密码")

if r.get('code') == 200:
    print("✅ SDK 正常")
    sys.exit(0)
elif r.get('code') == 1011:
    print("❌ 签名错误 — APP可能升级了，需要检查")
    sys.exit(1)
elif r.get('code') == 1012:
    print("⚠️ 请求过期 — 检查系统时间")
    sys.exit(2)
else:
    print(f"❓ 未知错误: {r}")
    sys.exit(3)
```

配合 cron 或 GitHub Actions 使用，失败时发通知。

## 七、历史经验：什么情况下需要完全重做

根据对同类 APP 的观察，以下情况需要从头来过：

| 变化 | 概率 | 重做工作量 |
|------|------|-----------|
| 换混淆工具 | 低 | 重写解码器，约2小时 |
| 换加密库 (CryptoJS→其他) | 极低 | 重新分析加密逻辑，约4小时 |
| 从 uni-app 换成原生 | 极低 | 完全重做，约2天 |
| 加了证书绑定/Root检测 | 中 | 绕过即可，约1小时 |
| 密钥改了但算法没变 | 中 | 提取新密钥，约30分钟 |
| 只改了版本号 | 高 | 改一行代码，约1分钟 |

**大部分升级只需要改 `APP_VERSION` 这一行。**
